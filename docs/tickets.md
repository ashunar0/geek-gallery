# 実装チケット一覧: GeekSalon 受講生作品集サイト

## Phase 1: 環境構築・セットアップ

- [ ] **#1 Neon DB セットアップ + Drizzle ORM 導入**
  - 概要: Neonプロジェクト作成、Drizzle ORMインストール、DB接続設定（環境変数）、スキーマファイル作成（users / works テーブル）、マイグレーション実行
  - 完了条件: `pnpm drizzle-kit push` でNeon上にテーブルが作成される。`.env.local` に `DATABASE_URL` が設定されている
  - 依存: なし

- [ ] **#2 Auth.js (NextAuth v5) + Google OAuth セットアップ**
  - 概要: Auth.jsインストール、Google OAuth クライアント作成、`auth.ts` 設定、`/api/auth/[...nextauth]` Route Handler 作成、セッション取得ユーティリティ作成
  - 完了条件: `/login` からGoogleログイン→セッション取得→ログアウトが動作する
  - 依存: #1（usersテーブルが必要）

- [ ] **#3 Cloudflare R2 接続セットアップ**
  - 概要: AWS SDK v3 インストール、R2バケット作成、S3Client設定、環境変数設定（`R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_ENDPOINT`, `R2_BUCKET_NAME`）
  - 完了条件: テスト用スクリプトでR2にファイルアップロード・取得ができる
  - 依存: なし

## Phase 2: 型定義・スキーマ

- [ ] **#4 型定義 + モックデータ作成**
  - 概要: `Work` / `User` のTypeScript型定義、Drizzleスキーマからの型推論設定、コース一覧の定数定義、モックデータ作成（作品5〜10件、ユーザー3件程度）
  - 完了条件: `types/` に型定義、`lib/mock-data.ts` にモックデータがあり、型エラーなし
  - 依存: #1（Drizzleスキーマ）

## Phase 3: フロントエンド実装（モックデータで動作）

- [ ] **#5 共通レイアウト + ヘッダー**
  - 概要: `app/layout.tsx` の共通レイアウト、ヘッダーコンポーネント（ロゴ、ログイン/ログアウトボタン、「投稿する」ボタン）、レスポンシブ対応
  - 完了条件: 全ページで共通ヘッダーが表示される。認証状態でボタン表示が切り替わる（モックで）
  - 依存: #4

- [ ] **#6 トップページ — 作品カード一覧**
  - 概要: `app/page.tsx` 作品カード一覧表示、`WorkCard` コンポーネント（サムネイル、作品名、制作者名、コース/期、技術スタックタグ）、コース別フィルタUI（タブ or セレクト）、新着順表示
  - 完了条件: モックデータで作品カードが一覧表示される。コースフィルタで絞り込みができる
  - 依存: #5

- [ ] **#7 作品詳細ページ**
  - 概要: `app/works/[id]/page.tsx` 作品の全情報表示（スクリーンショット大、作品名、URL外部リンク、技術スタック、説明文、制作者名→ユーザーページリンク、制作期間、こだわりポイント）、本人のみ「編集」ボタン表示
  - 完了条件: モックデータで作品詳細が全項目表示される
  - 依存: #5

- [ ] **#8 ユーザーページ**
  - 概要: `app/users/[id]/page.tsx` ユーザー情報（アバター、名前）+ そのユーザーの作品一覧（`WorkCard` を再利用）
  - 完了条件: モックデータでユーザー情報と作品一覧が表示される
  - 依存: #6（WorkCard再利用）

- [ ] **#9 投稿フォームページ**
  - 概要: `app/works/new/page.tsx` 入力フォーム（作品名、URL、スクリーンショットアップロード、技術スタック、説明文、制作者名、制作期間、コース/期セレクト、こだわりポイント）、バリデーション、画像プレビュー
  - 完了条件: フォームUIが表示され、バリデーションが動作する（送信はモックで `console.log`）
  - 依存: #5

- [ ] **#10 編集フォームページ**
  - 概要: `app/works/[id]/edit/page.tsx` 既存データを初期値にセットした編集フォーム（投稿フォームのコンポーネントを再利用）、削除ボタン + 確認ダイアログ
  - 完了条件: モックデータがフォームに初期表示される。削除確認ダイアログが動作する
  - 依存: #9（フォームコンポーネント再利用）

- [ ] **#11 ログイン / 新規登録ページ**
  - 概要: `app/login/page.tsx` と `app/register/page.tsx`。「Googleでログイン」ボタン、登録ページは同じUIだがコピー違い
  - 完了条件: ログイン/登録ページが表示され、ボタンクリックでAuth.jsのsignInが呼ばれる
  - 依存: #2（Auth.js）

## Phase 4: バックエンド実装

- [ ] **#12 作品 CRUD API**
  - 概要: `app/api/works/route.ts` (POST)、`app/api/works/[id]/route.ts` (PUT, DELETE)。Drizzleでのデータ操作、セッションによる認証チェック、本人のみ編集/削除の権限チェック
  - 完了条件: APIルートが動作し、認証・権限チェックが正しく機能する
  - 依存: #1, #2

- [ ] **#13 画像アップロード API**
  - 概要: `app/api/upload/route.ts`。R2への画像アップロード、ファイルサイズ制限（10MB）、形式チェック（jpg/jpeg/png/gif/webp）、image keyをレスポンスで返す
  - 完了条件: 認証済みユーザーが画像をアップロードし、R2に保存されたimage keyが返る
  - 依存: #2, #3

- [ ] **#14 作品一覧・詳細データ取得**
  - 概要: 作品一覧取得（コース別フィルタ、新着順ソート）、作品詳細取得、ユーザー別作品一覧取得のサーバーサイドデータ取得関数（`lib/queries/`）
  - 完了条件: RSCから直接データ取得ができる関数が動作する
  - 依存: #1

## Phase 5: フロントエンド ↔ バックエンド接続

- [ ] **#15 一覧・詳細ページをDB接続に切り替え**
  - 概要: トップページ / 作品詳細 / ユーザーページのデータソースをモックからDB取得に切り替え
  - 完了条件: Neon上の実データが画面に表示される
  - 依存: #6, #7, #8, #14

- [ ] **#16 投稿・編集フォームをAPI接続**
  - 概要: 投稿フォームの送信を `POST /api/works` に接続、画像アップロードを `POST /api/upload` に接続、編集フォームを `PUT /api/works/[id]` に接続、削除を `DELETE /api/works/[id]` に接続
  - 完了条件: フォームから作品を投稿→一覧に表示される。編集・削除が動作する
  - 依存: #9, #10, #12, #13

- [ ] **#17 認証状態によるUI出し分け**
  - 概要: ヘッダーのログイン/ログアウト切り替え、「投稿する」ボタンの表示制御、編集ボタンの本人のみ表示、未認証時のリダイレクト（`/works/new`, `/works/[id]/edit`）
  - 完了条件: 未ログイン/ログイン済み/本人で正しくUIが切り替わる
  - 依存: #2, #5, #11

## Phase 6: 仕上げ

- [ ] **#18 デプロイ + 環境変数設定**
  - 概要: Vercelプロジェクト作成、環境変数設定（DATABASE_URL, AUTH_SECRET, GOOGLE_CLIENT_ID/SECRET, R2関連）、本番デプロイ確認
  - 完了条件: Vercel上で本番サイトが動作する
  - 依存: #15, #16, #17
